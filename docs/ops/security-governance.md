# Security & Governance SOP (Standard Operating Procedure)

This document describes the security, consent, and release gate procedures for the `myvoiceclone` system.

## 1. 允许与禁止的用途 (Permitted and Prohibited Uses)

### Permitted Uses
- Cloning voices for which you have explicit, documented, and verified consent of the speaker.
- Local research and debugging in a secure, non-public environment.
- Evaluation of speech models and voice synthesis systems on authorized corpora.

### Prohibited Uses
- Voice cloning or speech synthesis of any speaker without explicit written consent.
- Generating synthetic media (deepfakes) for impersonation, harassment, scamming, or spreading misinformation.
- Releasing models publicly without passing the local release gate checks or receiving an explicit authorization waiver.

---

## 2. 授权凭证录入 (Consent Evidence Registration)

Consent must be explicitly recorded in the `consent_ledger` database table before a voice model can be released.

### Database Insertion Example
To register consent for a speaker:
```sql
INSERT INTO consent_ledger (id, speaker_id, recording_id, granted, signature)
VALUES (
    'consent_uuid_1', 
    'speaker_id_123', 
    'recording_id_456', 
    1, -- 1 = granted, 0 = denied/revoked
    'cryptographic_signature_placeholder_or_hash'
);
```

## 3. 发布闸处理 (Release Gate Operations)

Every trained model run must go through a release gate check before its outputs can be distributed.

### Release Gate Lifecycle States
- **Pending**: Gate has been created but checks have not run or completed.
- **Passed**: Consent check passed (all speakers in the dataset have granted consent).
- **Failed**: Consent check failed due to unauthorized speakers.
- **Waived**: The failure was overridden with an explicit reason.

### Creating a Release Gate (via API)
```http
POST /api/reports/release-gates
Content-Type: application/json

{
  "gate_id": "gate_123",
  "model_run_id": "run_456"
}
```

### Waiving a Failed Gate
If a gate fails because of a missing signature but the developer holds physical evidence, they can waive it by providing an explicit reason:
```http
POST /api/reports/release-gates/gate_123/waive
Content-Type: application/json

{
  "approved_by": "supervisor_alice",
  "reason": "Physical consent form verified on file and archived."
}
```
> [!IMPORTANT]
> A waive request MUST contain a non-empty `reason` string, which will be logged permanently in the `policy_events` table for audit trace.

---

## 4. P0-P6 与 P7 安全边界 (P0-P6 and P7 Security Boundaries)

To ensure rapid developer iterations, the security controls follow a post-retrofit model:
- **P0-P6 Pipelines (Diarization, Separation, Transcription, Feature Extraction, Model Training, Evaluation)**: These pipelines do NOT intercept or block executions. This allows models to be trained and evaluated locally without any setup overhead.
- **P7 Release Gate & Metadata Retrofit**:
  - The security validation is enforced at the **release stage** (i.e. creating release gates).
  - Synthetic outputs generated by inference are tagged with metadata (`synthetic=True`, `source_model_run`, and `watermark="placeholder"`) to guarantee trace lineage.
  - Security policies can be completely enabled/disabled via the `security.enabled` configuration flag in `configs/local.yaml`.
